# Chess research status — 2026-07-13

This document is a detailed handoff for the current sub-50M-parameter Chess GM research thread. It records the working hypothesis, implemented architecture, experiment evidence, active run, unresolved questions, operational commands, and intended next steps as of commit `96e81e2`.

## Executive summary

The project has cleared two early proof-of-life milestones:

1. A compact Q-Former history encoder can learn nontrivial chess/state-related facts from tokenized move history.
2. The frozen encoder interface preserves enough information for a separate inverse-transition decoder to begin reconstructing the immediate next move from `state_t` and `state_t+1`; a random-encoder control is still required to attribute that result to learned representation quality.

The most important recent architectural change is the removal of mean pooling from the active Q-probe path. Each probe feature now has its own learned `[1, D]` query, single-head cross-attention over the full Q-Former bank, MLP, and classifier. Early training curves from the live `expert-disco-13` run are consistent with better sample efficiency than the old mean-pooled probe. This remains a provisional result because the heads are larger, the comparison is not held out, and the runs are not fully matched.

The immediate research direction is now clearer:

```text
move-history trajectory
    → Q-Former state bank
    → task-specific questions of that state bank
    → explicit state-feature supervision
    → tactical and strategic future-state prediction
    → inverse transition to an immediate move
```

The next encoder intended for the full architecture should be grounded with explicit board-state queries, not only check/mate/turn or WDL labels. The proposed `DiffThinkerBoardStateQueryMLP` will ask questions such as “what occupies e4?” using a square-query embedding and a shared occupant response space. If it can reconstruct held-out positions from randomly sampled square queries, the Q-bank becomes a credible, queryable world-state representation for later planning.

No claim of chess strength has been established. The current evidence concerns representation and transition decoding, which is the intended scope at this stage.

## Research thesis

The model is intended to represent and reason over trajectories, not merely imitate the next move autoregressively.

Move history has a useful causal form:

```text
I do this
→ the opponent does that
→ I do this
→ the opponent is constrained
→ this ends in mate / conversion / a favorable state
```

A board snapshot is useful for legality and explicit state, but move history preserves the action narrative needed for trajectory reasoning. The current direction is therefore to ground the move-history encoder with explicit board-state supervision rather than immediately replace it with a board-only representation.

The longer-term conceptual hierarchy is:

```text
z_t = Q-encoded current trajectory/state

strategy thinker:
    z_t → desirable z_(t+H)
    “What state should exist several moves from now?”

tactical thinker:
    (z_t, desirable z_(t+H)) → proposed z_(t+1)
    “What immediate successor moves toward that strategy?”

inverse-transition decoder:
    Q = proposed z_(t+1)
    K,V = z_t
    → immediate move packet
```

The eventual strategic/tactical thinker is expected to explore diffusion or flow-style latent-state generation. A value/critic model will be required to distinguish plausible futures from desirable futures. Delayed reward and RL remain later research problems.

### Move packets

Each ply is represented as a fixed packet of eight compositional tokens. Examples include turn, piece, source disambiguation, capture, destination, promotion, check/mate, `<EOM>`, and padding. The vocabulary currently has 108 tokens.

Conceptual shape:

```text
moves.npy: uint16[total_plies, 8]
offsets.npy: int64[num_games + 1]
results.npy: int64[num_games]
```

Result labels:

```text
0 = white win
1 = black win
2 = draw
```

All eight token slots in a ply receive the same RoPE ply position. Position increments by ply, not by token slot.

## Current encoder architecture

### History encoder

`PlyHistoryEncoder` receives `[B, T, 8]`, embeds the eight-token ply packets, flattens them to `[B, T*8, D]`, and applies causal transformer blocks.

The attention mask combines:

- a key-padding mask, preventing attention to `<PAD>` keys;
- a causal ply mask, preventing attention to future plies;
- same-ply visibility, allowing all eight slots in one ply to interact.

The current architecture uses relative positions within the cropped context. A context cropped from late in a game is positioned from zero again; absolute game ply is not currently encoded.

### Q-Former bottleneck

`QFormerPlyHistoryEncoder` cross-attends a fixed bank of learned queries to the variable-length encoded move history:

```text
input history: [B, T, 8]
history tokens: [B, T*8, D]
Q-Former output: [B, K, D]
```

Current main probe configuration:

```text
D = 256
K = 16 learned queries
history layers = 4
Q-Former cross-attention layers = 2
heads = 16
```

The reusable encoder has 4,770,304 parameters under this configuration.

### Task-specific `DiffThinkerMLP` heads

The active Q-probe no longer mean-pools the Q-bank. It has three independent task heads:

```text
check head:
    learned q_check [1, D]
    → single-head cross-attention over Q-bank
    → residual MLP
    → 2-class classifier

mate head:
    learned q_mate [1, D]
    → independent cross-attention/MLP/classifier

turn head:
    learned q_turn [1, D]
    → independent cross-attention/MLP/classifier
```

Each learned query acquires meaning through its task loss:

```text
q_check: “Does this state encode check?”
q_mate:  “Does this state encode mate?”
q_turn:  “Whose turn follows this state?”
```

Each current head has 791,554 parameters. The complete current Q-probe has 7,144,966 parameters:

```text
encoder:      4,770,304
check head:     791,554
mate head:      791,554
turn head:      791,554
total:        7,144,966
```

This is substantially larger than the earlier ~4.97M mean-pooled probe because every task now owns a full cross-attention/MLP head. This is an intentional feature-learning architecture, but it is also a capacity confound that must be controlled in later ablations.

## Probe supervision

The Q-probe samples a random prefix and predicts:

- whether the final sampled ply resulted in check or mate;
- whether it resulted specifically in mate;
- whose turn is next.

For this diagnostic probe only, `CHECK` and `MATE` tokens are removed from every input ply and remaining packet tokens are shifted left. Therefore the model cannot read those labels directly. In the eventual production trajectory representation, check and mate tokens are expected to remain normal encoded inputs.

The turn probe is structurally easy because turn follows ply parity and can also be exposed by context/padding structure. It is useful as a readout sanity check, not proof of deep chess representation.

CHECK and MATE are more meaningful but highly imbalanced. The live `expert-disco-13` run started under the older CHECK positive-class schedule:

```text
initial positive weight: 20
final positive weight: 1
linear decay: 150,000 batches
```

The repository default has since changed to:

```text
initial positive weight: 5
final positive weight: 1
linear decay: 150,000 batches
```

This affects only future invocations unless the current run is restarted. Weighted CHECK precision should not be interpreted without recall and prediction counts. The console/W&B metrics now include:

```text
p_check
r_check
check_positive / chk+
check_pred_positive / chk_pred
check_tp / chk_tp
```

## Experiment history and evidence

### 1. Mean-pooled Q-probe proof of life

The first Q-probe used:

```text
Q-Former bank [B, K, D]
→ mean over K
→ separate check/mate/turn MLP heads
```

Three local snapshots survive from a prior two-epoch run:

```text
q_probe_epoch_002_batch_005000.pt  global batch 181,450
q_probe_epoch_002_batch_010000.pt  global batch 186,450
q_probe_epoch_002_batch_015000.pt  global batch 191,450
```

All model tensors changed between snapshots, so the run was actively learning. It stopped during epoch 2; no final epoch-2 checkpoint exists locally.

The old run established:

- turn could be learned through the Q-bottleneck;
- CHECK loss fell substantially despite removal of explicit CHECK/MATE input tokens;
- CHECK precision eventually rose to roughly 0.65–0.70 as the positive weight decayed toward 1;
- the compact encoder carried nontrivial tactical/move-state information.

The correct conclusion was proof of representation life, not chess strength or full board understanding.

### 2. Frozen-encoder inverse-transition proof of life

`QInverseTransitionDecoder` asks:

```text
z_before = encoder(history through state_t)
z_after  = encoder(history through state_t+1)

cross-attention:
    Q = z_after
    K,V = z_before

→ transition query bank
→ eight learned move-slot queries
→ [B, 8, vocab] simultaneous move-packet logits
```

There is no mean pooling. Direct highways from both state banks feed the transition stack. All eight output slots are predicted simultaneously, not autoregressively.

A frozen copy of the old trained encoder was used with a 16-ply context. Early training metrics progressed approximately as follows:

```text
batch 100:  loss 2.889, non-PAD token accuracy 0.019, packet accuracy 0
batch 1000: loss 1.845, non-PAD token accuracy 0.252, packet accuracy 0.0003
batch 2000: loss 1.042, non-PAD token accuracy 0.507, packet accuracy 0.0098
batch 2900: loss 0.978, non-PAD token accuracy 0.521, packet accuracy 0.0143
```

This shows that a new decoder can extract immediate-move information through the frozen encoder interface. It is not yet a held-out evaluation and does not isolate the value of encoder pretraining: because `state_t+1` contains the target move, an untrained but information-preserving encoder may also support reconstruction. A fresh/random-encoder control is required. This experiment is explicitly an inverse-dynamics/interface test, not proof of planning.

The decoder model is approximately 8.1M parameters at the current 256-wide configuration.

### 3. Task-query Q-probe: live `expert-disco-13` run

It uses the new independent task-query heads and, because it began before the default changed, appears to use the old 20→1 CHECK positive-weight schedule. ## Inverse-transition data curriculum

The inverse-transition trainer now has a more detailed outcome-conditioned sampler. It defaults to 20,000,000 sampled transitions per epoch, sampled with replacement.

The intended rule is:

- for a White move target, sample from a White-win game;
- for a Black move target, sample from a Black-win game;
- exclude draws;
- sample target transition locations up to approximately 200 plies.

The implementation pre-indexes games by result and length so compatible-game selection does not scan the full corpus per example.

This is a weak “move in the eventual winner’s direction” training signal, not a value model. It is currently secondary to validating and grounding the encoder.

## Proposed full encoder: `DiffThinkerBoardStateQueryMLP`

The current check/mate/turn heads validate the query-readout interface. The encoder intended for the full architecture should receive explicit board-state grounding.

### Core query design

Use one shared square-query module, not 64 unrelated heads:

```text
square query embedding:       E_square[64, D]
occupant response embedding:  E_occupant[13, D]

query(square=e4) + Q-bank(state)
→ cross-attention
→ MLP
→ occupant response
```

Occupant classes:

```text
EMPTY
WHITE_PAWN, WHITE_KNIGHT, WHITE_BISHOP, WHITE_ROOK, WHITE_QUEEN, WHITE_KING
BLACK_PAWN, BLACK_KNIGHT, BLACK_BISHOP, BLACK_ROOK, BLACK_QUEEN, BLACK_KING
```

Do not use redundant classes such as `WHITE_KNIGHT_ON_WHITE`. Square color is already determined by the square query.

The model should accept multiple square queries:

```text
queries: [B, Q, D]
state bank: [B, K, D]
output: [B, Q, 13]
```

Sample a subset of squares per training position for efficiency; query all 64 at evaluation to reconstruct the complete board.

### Why this matters

This turns the architecture into a queryable world model:

```text
query(object/location) + latent world state → predicted attribute/state
```

The chess implementation is domain-specific, but the mechanism generalizes directly to robotics-style object, joint, pose, contact, or waypoint queries.

Balanced query sampling can avoid the severe imbalance of CHECK. Training can deliberately sample occupancy classes or occupied squares while retaining a mixture of uniform random-square queries for natural-distribution calibration.

### Data generation

Replay accepted PGNs with `python-chess` during the same preprocessing pass that writes the game store. After each pushed ply, record the 64-square occupancy aligned with that ply row.

Thirteen classes fit in four bits. A compact proposed store is:

```text
board_after_packed.npy: uint8[total_plies, 32]
```

Two square labels are packed per byte and unpacked only for sampled examples.

A separate replay pass is risky unless exact game filtering and order alignment can be guaranteed.

Occupancy is not the complete chess state. Later explicit queries/targets should include:

- side to move;
- castling rights;
- en-passant square;
- halfmove/repetition-relevant state if needed.

### Promotion criteria

Before treating this encoder as the foundation for tactical/strategic planning, evaluate on held-out games:

- random sampled-square occupancy accuracy;
- per-piece/class accuracy;
- full 64-square reconstruction accuracy;
- complete-position exact match or square-error distribution;
- side/castling/en-passant accuracy when implemented;
- frozen inverse-transition decoding versus a fresh/random encoder.

Numerical promotion thresholds have not been chosen yet. They should be set before the large run, after measuring trivial baselines (always empty, class-prior, direct replay sanity checks) and a fresh/random encoder. At minimum, promotion requires a held-out advantage over those controls and no collapse on rare piece classes; training accuracy alone is insufficient.

## Bottleneck hypothesis

A planned ablation tests whether a fixed narrow representation bottleneck encourages better generalization.

Proposed architecture:

```text
wide history transformer
        ↓
fixed Q-Former bank [K, D]
        ↓
wide task-query/classifier heads
```

The claim cannot be tested by widening only one model and observing improvement. At minimum compare:

1. current encoder + current heads + fixed Q-bank;
2. wide encoder + wide heads + the same Q-bank width/capacity;
3. wide encoder + wide heads + a wider Q-bank.

Keep data and evaluation fixed. A particularly useful test is to freeze each learned Q-bank encoder and train the same small probe on top. If the wide–narrow model’s representation performs better through the same small downstream probe, that is stronger evidence for the bottleneck hypothesis than end-to-end task accuracy alone.

## What is established versus not established

### Established enough to proceed

- The tokenizer/data pipeline works at multi-million-game scale.
- The Q-Former bottleneck is trainable.
- The compact encoder learns at least some state/tactical information without explicit CHECK/MATE leakage.
- The old frozen encoder supports early inverse-transition learning.
- Task-specific query heads show a large early turn-learning advantage and a smaller early CHECK advantage in the currently observed training curves.
- The whole current probe remains small at ~7.15M parameters; the inverse-transition model is ~8.1M.

### Promising but not yet proven

- That the new head causes the Q-Former itself to learn better features, rather than only providing a stronger extractor.
- That CHECK gains persist through the full epoch and onto held-out games.
- That the Q-bank contains a sufficiently complete board representation.
- That outcome-conditioned transition sampling produces stronger moves rather than population imitation.
- That a narrow Q-bank improves generalization at fixed upstream/downstream scale.

### Not established

- Legal move generation.
- Chess playing strength.
- A reliable value/critic model.
- Strategic future-state generation.
- Diffusion/flow-based thinking.
- RL with delayed reward.
- GM-level performance.

## Known caveats and technical debt

1. **No held-out evaluation loop yet.** Most current evidence is from training curves. Game-level train/eval splits are required before making representation-generalization claims.
2. **Head-capacity confound.** The new three-head probe is ~7.15M versus ~4.97M for the old probe. A fair ablation needs task-query + linear/small-MLP variants and matched parameter controls.
3. **Probe epoch sampling is deterministic by item index.** The current dataset seeds each sample from `seed + idx`; without changing the seed by epoch, later epochs revisit the same sampled game/prefix mapping in a different loader order.
4. **No faithful probe resume.** Probe checkpoints contain model and optimizer state, but `--init-checkpoint` loads compatible encoder weights only. It does not resume heads, optimizer, epoch, batch, or global batch.
5. **CHECK/MATE imbalance.** Weighted loss changes the precision/recall operating point. Always inspect precision, recall, positive targets, predicted positives, and true positives together.
6. **Turn is structurally easy.** It validates readout optimization but should not be treated as evidence of strategic understanding.
7. **Quadratic history attention.** The history encoder attends over `context_plies × 8` flattened tokens and is run twice for inverse transitions. Long contexts can become expensive even with a frozen encoder.
8. **Relative cropped positions only.** Absolute game ply is not represented after context cropping.
9. **Board-state supervision is only designed.** `python-chess` replay, packed labels, board-query datasets, heads, and metrics are not implemented.
10. **Legacy WDL path remains.** `QVerifierTransformer` still mean-pools.
11. **Current run configuration must be read from W&B.** The live run began before later defaults/checkpoint naming changes; repository defaults do not retroactively alter it.
12. **The external corpus is not pinned.** Archive inventory/checksums and the exact 5.65M-game manifest have not been checked into a durable experiment record.
13. **No numerical decision gates yet.** Board-query and bottleneck promotion criteria are listed, but thresholds must be fixed against baselines before those experiments begin.

## Bottom line

The project is still early, but it is no longer architecture-free speculation. A compact Q-Former state bank has supported targeted fact extraction, and its interface supports early frozen inverse-transition learning. Task-specific learned questions are showing materially better early training behavior than mean pooling, subject to the unresolved capacity and evaluation controls documented above.

The highest-value next step is to turn that interface into explicit, randomly queried board-state supervision. If the encoder can answer arbitrary square-occupancy questions and preserve transition-decodable information on held-out games against fresh-encoder controls, it will be a strong foundation for the intended trajectory-planning system.
